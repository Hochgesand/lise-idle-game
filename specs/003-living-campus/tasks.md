# Tasks: Living Campus — Player Character, Activities on the Map, Visible Timed Jobs

**Input**: Design documents from `/specs/003-living-campus/`

**Prerequisites**: plan.md (required), spec.md (required), research.md, data-model.md, quickstart.md — no `contracts/` (this feature adds no REST/STOMP surface; the sim-internal shapes are fixed in data-model.md)

**Tests**: **MANDATORY** — the project constitution (`.specify/memory/constitution.md` Principle III "Test-First — NON-NEGOTIABLE") requires TDD: tests written, approved, and RED before implementation. Test tasks are included and MUST be completed before their sibling implementation tasks in every phase. Frontend suite baseline at feature start: **549 tests / 29 files, all green** (verified 2026-07-02, `cd frontend && npx vitest run`) — the FULL suite stays green at every GREEN commit.

**Deploys**: **Every phase ends with an explicit "Deploy to prod & verify" task** executed via the `deploy-lise-game` Claude Code skill (created in 002 T008; GHCR pull-based per the 002-refined procedure). The live site at `https://lise-game.schmitz.gg` is re-proven after each phase — each phase Checkpoint references its deploy task, and each deploy task names the quickstart.md scenarios it must run.

**Organization**: Tasks are grouped by user story to enable independent implementation and testing of each story. See spec.md for the three user stories (US1 P1 player character, US2 P2 activities on the map, US3 P3 visible timed jobs). The Foundational phase carries the feature's cross-cutting save/content/backend groundwork (schema v3, `world` content block, passthrough) so every story phase deploys against a backend that already speaks v3.

## Phase 5: User Story 3 - Visible Timed Jobs, CoC-lite (Priority: P3)

**Goal**: Running jobs render as in-world progress indicators above their stations (pure derivations of state + elapsed time); trainings gain data-driven durations — the feature's only sim change: `startTraining` (cost-at-start, one-at-a-time, zero-duration = instant) supersedes `purchaseTraining`, and `advance` resolves completion at `startedAt + durationSeconds` inside the piecewise machinery, deterministic across offline spans.

**Independent Test**: quickstart Scenarios 8–10 — an active burner run shows a draining indicator above its station; starting a timed training deducts the cost immediately, shows an Academy indicator, occupies the avatar, never hard-gates other actions, and applies its multiplier only from completion; closing the game mid-training and reopening past the end yields the exact offline credit an online session would have produced.

## Implementation Strategy

## Notes

- **TDD is mandatory** (Constitution Principle III, non-negotiable): every test task must be RED before its sibling implementation. The frontend baseline at feature start is 549 tests / 29 files — the FULL suite stays green at every GREEN commit.
- **The sim boundary is structural**: only `activeTraining` (state) and `durationSeconds` (content) enter the sim; `WalkIntent`, find-me, indicators, and the player projection are views (data-model §9 is the enforceable checklist). Nothing in `frontend/src/sim/` may import from `scenes/` or read a wall clock; `world.walkSeconds` is consumed only by the presentation layer.
- **The two plan-tracked borderline items get explicit test guardrails**: the wall-clock walk timer (T026's state-equality assertions + the FR-013 fallback menu ensuring no capability is only reachable through the timed path) and the timed-training sim change (T040's associativity + large-`dt` offline property tests).
- **Offline/signed-out completeness** (FR-023): every 003 affordance must be demonstrable in an anonymous, offline session — checked at every deploy task and in full at T055 (Scenario 11); the label is the only identity-aware pixel, with "Du" fallback.
- **No new backend surface**: content files + state passthrough + schema bump only (T005/T006/T008/T011–T013). Anything needing a new endpoint is out of scope by spec.
- Big numbers stay strings end-to-end; durations are bounded plain scalars (seconds), never resources.
- Commit after each task or logical group; work lands on `main` per the repo's trunk convention.
